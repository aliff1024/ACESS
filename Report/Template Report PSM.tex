\textbf{ABSTRACT}
\end{center}

\vspace{1cm}

\begin{spacing}{1.1}
Traditional e-learning platforms are predominantly designed with a one-size-fits-all approach, presenting significant accessibility barriers for neurodivergent learners, particularly those with Dyslexia, Attention Deficit Hyperactivity Disorder (ADHD), and Autism Spectrum Disorder (ASD). These systems typically feature dense typography, complex navigational structures, and rigid content delivery that contribute to cognitive overload and high dropout rates. This project presents the development of the Adaptive Cognitive and Educational Skill Support (ACESS) platform, a web-based learning management system engineered to dynamically accommodate diverse cognitive profiles through natively integrated accessibility features. The system was developed using the Agile Scrum methodology over six iterative sprints, employing Next.js with the App Router for server-rendered React components, Supabase for backend-as-a-service with PostgreSQL Row-Level Security, and Tailwind CSS v4 for dynamic theme injection. ACESS implements core features including a rule-based adaptive engine that automatically recommends remedial content based on assessment performance, configurable accessibility profiles that alter typography, spacing, and motion preferences in real time via CSS custom properties, a strict Focus Mode for learners with attention-related challenges, and comprehensive analytics dashboards for educators to monitor engagement and identify at-risk students. The evaluation methodology comprised structured walkthroughs guided by task scenarios for each learner profile (Dyslexia, ADHD, ASD), complemented by an expert review conducted against WCAG 2.1 Level AA success criteria and established usability heuristics to systematically assess the platform's accessibility compliance. These evaluations demonstrated that the adaptive engine and accessibility settings effectively reduce cognitive friction and accommodate diverse learner needs compared to conventional platforms. The findings indicate that a purpose-built, architecturally adaptive system can significantly reduce cognitive friction and promote equitable educational outcomes for learners with diverse neurological needs.
\end{spacing}

\section{INTRODUCTION}
\end{center}
\vspace{2cm}

\subsection{Background of Study}

In the current educational landscape, instruction is delivered through a blend of traditional face-to-face classroom settings and online platforms \cite{prensky2001digital}. While digital tools have become increasingly common to support learning, many institutions still rely heavily on traditional teaching methods. This hybrid approach presents unique challenges for neurodivergent individuals, specifically those with Dyslexia, Attention Deficit Hyperactivity Disorder (ADHD), and Autism Spectrum Disorder (ASD). Whether learning occurs in a physical classroom or online, the tools used to deliver content often lack the flexibility required to accommodate diverse cognitive profiles.

Existing Learning Management Systems (LMS) are typically designed with a ``one-size-fits-all'' approach. They present dense textual information, complex navigation structures, and rigid assessment formats. These standard interfaces can erect significant barriers for neurodivergent learners. For instance, learners with Dyslexia may struggle with standard typography and line spacing \cite{franceschini_dyslexia}, while those with ADHD might find cluttered interfaces and unstructured learning paths overwhelming \cite{barkley1997adhd}. 

Therefore, there is a critical need for a natively adaptive learning platform that can seamlessly support both face-to-face and remote learning environments. The Adaptive Cognitive and Educational Skill Support (ACESS) platform is proposed to bridge this gap. ACESS is engineered to provide an inclusive digital learning environment that dynamically alters its interface and content delivery based on the specific cognitive requirements of the user. By integrating features such as dynamic typography adjustment, focus-driven layouts, and rule-based adaptive learning paths, ACESS aims to ensure that educational materials are accessible and equitable for all learners, regardless of the instructional setting.

\subsection{Problem Statement}

Despite the availability of various educational technologies, significant cognitive accessibility barriers persist. The core problem this project addresses is the systemic inadequacy of current educational platforms to serve learners with disabilities.

Specifically, the problems identified are:
\begin{enumerate}
    \item Most existing e-learning platforms are not designed for learners with disabilities. They use formats that overwhelm cognitive and sensory processing, leading to high dropout rates. Research indicates that learners with disabilities face significantly higher attrition rates in online education; for instance, only 46\% of neurodivergent students completed online courses compared to 74\% of their neurotypical peers \cite{spencer2022neurodiversity}, while up to 35\% of students with ADHD drop out of e-learning programs within the first semester due to poor accessibility design \cite{burl2024neurodiversent}.
    \item Learners with disabilities lack platforms that adapt to their individual learning pace and present content in accessible, easy-to-process formats.
    \item Employers and support organisations have no reliable way to verify the skills gained by learners with disabilities due to the absence of proper progress tracking and accessible certification.
    \item Educators and mentors do not have a centralised platform to monitor learner progress, identify those who need extra support, and manage accessible learning materials effectively.
\end{enumerate}

\subsection{Objectives}

This project aims to achieve the following objectives:
\begin{enumerate}
    \item To analyze the accessibility requirements, learning challenges, and content delivery needs of learners with dyslexia, ADHD, and Autism Spectrum Disorder (ASD) through literature review and comparative study of existing e-learning platforms and assistive technology standards.
    \item To develop a fully functional accessible web-based e-learning platform that incorporates rule-based adaptive content delivery, built-in Text-to-Speech (TTS) support, adjustable accessibility display settings, structured lesson and quiz management, and role-based user management for learners, educators, and administrators.
    \item To implement a certificate generation module and an educator analytics dashboard that provide verifiable skill completion records and enable data-driven monitoring of individual learner progress.
    \item To evaluate the functional correctness, usability, and accessibility of the developed platform through structured walkthroughs and expert review.
\end{enumerate}

\subsection{Scope}

The scope of the ACESS platform defines the boundaries of the system architecture, target users, and core functional modules. The system is designed to act as a supportive educational tool for both traditional and online learning environments.

\subsubsection{Target Users}
\begin{itemize}
    \item \textbf{Learners:} The primary consumers of the educational content. The interface is specifically optimized for neurodivergent individuals, though it remains highly beneficial for neurotypical users.
    \item \textbf{Educators:} The course creators and administrators who require structured, intuitive tools to author accessible content, manage media assets, and monitor student progress.
    \item \textbf{Administrators:} System operators responsible for platform governance, user role management, and global accessibility configuration.
\end{itemize}

\subsubsection{Core System Modules}
The ACESS architecture is divided into the following core modules:
\begin{itemize}
    \item \textbf{Authentication and Profile Module:} Manages secure registration, login, role-based access control for learners, educators, and administrators, email verification, password reset, and profile management.
    \item \textbf{Accessibility and Personalization Engine:} The central differentiator of ACESS. Stores per-user accessibility preferences defining cognitive profiles with granular controls over font, spacing, theme, focus mode, reduced motion, and text-to-speech. Settings are applied globally, requiring no manual configuration after onboarding.
    \item \textbf{Educator Authoring Module:} Provides an interface for creating structured courses with chapters and milestones, writing lessons, uploading video assets with time-coded questions, designing interactive activities, managing lesson version history, and configuring quiz, certificate, and achievement criteria.
    \item \textbf{Adaptive Learning Module:} Governs course enrollment, lesson progression, dynamic UI alterations based on accessibility profiles, and personalized recommendations triggered by quiz performance, including a rule-based engine that suggests revision, standard, or advanced content.
    \item \textbf{Interactive Assessment Module:} Handles timed and untimed quiz delivery, multiple attempts, automated grading, answer review, and pass/fail threshold evaluation that triggers remedial recommendations when scores fall below configured thresholds.
    \item \textbf{Analytics and Telemetry Module:} Tracks engagement metrics such as time per lesson, completion rates, and scores, and provides educators with per-student dashboards with at-risk detection, plus platform-wide analytics for administrators.
    \item \textbf{Certificate and Achievement Module:} Generates personalized PDF certificates upon course completion and administers a badge system that awards visual milestones based on predefined criteria to enhance motivation.
    \item \textbf{Interactive Content Module:} Supports multiple activity types with dedicated builder and viewer components, plus time-coded video questions and structured content blocks.
    \item \textbf{Communication and Notification Module:} Manages threaded lesson discussions, a contact form for inquiries, and a database-triggered notification system that alerts users on enrollment, progress, quiz attempts, lesson additions, and course publication.
\end{itemize}

\subsection{Significance of Project}

The development of the ACESS platform represents a significant step towards educational equity. By shifting the burden of accessibility from the user to the system architecture, the platform ensures that neurodivergent learners can focus entirely on comprehending the educational material rather than struggling with the interface.

For learners, the project provides a calm, predictable, and highly customizable environment that mitigates anxiety and cognitive fatigue. For educators, it offers a streamlined toolset that enforces accessibility best practices during content creation, alongside deep analytical insights into student engagement. Ultimately, ACESS demonstrates how modern web architecture can be leveraged to create a truly inclusive educational ecosystem applicable in both physical classrooms and remote learning scenarios.

\subsection{Expected Output}

The ACESS platform is expected to produce the following deliverables upon completion:

\begin{itemize}
    \item \textbf{Adaptive Web Application:} A fully functional, production-ready learning management system accessible via standard web browsers and capable of dynamically adjusting its interface based on user accessibility profiles.
    \item \textbf{Role-Based Dashboards:} Three distinct interfaces for Learners, Educators, and Administrators, each with tailored navigation, permissions, and tooling appropriate to their respective responsibilities.
    \item \textbf{Accessibility Customization System:} A comprehensive set of user-configurable accessibility presets covering cognitive profiles, including typography control, theme switching, focus mode, text-to-speech, and reduced motion toggles.
    \item \textbf{Adaptive Recommendation Engine:} A rule-based engine that automatically analyzes quiz performance and generates personalized lesson recommendations to address identified knowledge gaps.
    \item \textbf{Course Authoring Tools:} An editor integrated with media upload, interactive activity builders, and content embedding for educators.
    \item \textbf{Quiz and Assessment System:} A complete assessment lifecycle including timed and untimed quizzes, automated grading, multiple attempt management, and answer review capabilities.
    \item \textbf{Analytics Dashboard:} Visual engagement metrics and progress tracking with per-student and per-course views for educators, plus platform-wide analytics for administrators.
    \item \textbf{Certificate and Achievement System:} Automated PDF certificate generation upon course completion and a badge-based achievement system to incentivize learner progress.
    \item \textbf{Notification System:} Database-triggered real-time notifications for enrollment, quiz results, lesson additions, and course publication across all user roles.
\end{itemize}

\subsection{Conclusion}

This chapter established the foundational context for the ACESS project by identifying the critical accessibility barriers present in current educational platforms, defining the specific problems to be addressed, and outlining the objectives and scope of the proposed system. The significance of the project was discussed in terms of its potential impact on educational equity for neurodivergent learners, and the expected deliverables were enumerated. The next chapter presents a comprehensive literature review, examining existing theoretical frameworks, analyzing current Learning Management Systems, describing the project methodology, and detailing the software and hardware requirements necessary to realize the ACESS platform.

\section{LITERATURE REVIEW AND PROJECT METHODOLOGY}
\end{center}
\vspace{2cm}

\subsection{Introduction}

The literature review provides the academic and technical foundation upon which the ACESS platform is conceptualized. This chapter synthesizes contemporary research regarding cognitive accessibility in digital education, evaluates established theoretical frameworks such as Universal Design for Learning (UDL) \cite{udl_guidelines}, and rigorously analyzes existing Learning Management Systems (LMS). By comparing platforms like Moodle \cite{moodle_software}, Canvas \cite{canvas_lms}, Blackboard \cite{blackboard_learn}, and Google Classroom \cite{google_classroom} against the specific requirements of neurodivergent learners, this chapter identifies the critical systemic gaps that justify the development of a natively adaptive, rule-based LMS architecture.

\subsection{Theoretical Frameworks for Cognitive Accessibility}

The design of digital interfaces for educational purposes must be grounded in empirically validated frameworks. The two primary pillars supporting the ACESS architecture are the Universal Design for Learning (UDL) guidelines \cite{udl_guidelines} and the Web Content Accessibility Guidelines (WCAG) 2.2 \cite{wcag22}.

\subsubsection{Universal Design for Learning (UDL)}
Universal Design for Learning is a framework developed by CAST \cite{udl_guidelines} aimed at optimizing teaching and learning for all individuals based on scientific insights into how humans learn. UDL is predicated on three primary principles:
\begin{itemize}
    \item \textbf{Multiple Means of Engagement:} Recognizing that learners differ markedly in the ways they are motivated. For neurodivergent learners, particularly those with ADHD, motivation can be severely hampered by unstructured environments \cite{barkley1997adhd}. ACESS implements this by offering gamified achievements and structured, step-by-step progressive disclosure of course material to prevent cognitive overwhelm.
    \item \textbf{Multiple Means of Representation:} Learners vary in how they perceive and comprehend information. A Dyslexic learner may struggle with printed text but excel with audio. ACESS implements this principle via integrated Text-to-Speech (TTS) capabilities and dynamic CSS injection that allows the user to radically alter the typography (e.g., using Atkinson Hyperlegible \cite{atkinson2021atkinson}) and contrast ratios.
    \item \textbf{Multiple Means of Action and Expression:} Learners differ in how they navigate a learning environment and express what they know. ACESS supports this by providing multiple quiz formats (scenario-based, multiple-choice) and allowing untimed assessments to reduce anxiety-induced performance degradation.
\end{itemize}

\subsubsection{Web Content Accessibility Guidelines (WCAG) 2.2}
While UDL provides the pedagogical framework, WCAG 2.2 \cite{wcag22} provides the strict, measurable technical parameters for digital accessibility. The ACESS platform targets Level AA compliance. Key WCAG criteria directly addressed by ACESS include:
\begin{itemize}
    \item \textbf{Guideline 1.4.12 (Text Spacing):} The ability to override default text spacing (line height, letter spacing, word spacing) without loss of content or functionality. This is a critical requirement for Dyslexia, which ACESS solves via its Tailwind CSS \texttt{@theme} variable injection.
    \item \textbf{Guideline 2.2.1 (Timing Adjustable):} Providing users the ability to turn off, adjust, or extend time limits. ACESS allows educators to configure untimed quizzes specifically to accommodate learners requiring extended processing time.
    \item \textbf{Guideline 2.3.3 (Animation from Interactions):} Motion animation triggered by interaction can be disabled. The ACESS system implements a global attribute that globally disables all animations, critical for learners prone to vestibular disorders or severe distraction (ADHD).
\end{itemize}

\subsection{Neurodivergent Profiles and Interface Interventions}

Understanding the specific cognitive challenges faced by the target demographic is essential for architecting effective UI/UX interventions.

\subsubsection{Attention Deficit Hyperactivity Disorder (ADHD)}
ADHD is characterized by executive dysfunction \cite{zelazo2003executive}, impacting working memory, task initiation, and sustained attention \cite{barkley1997adhd}. In digital environments, learners with ADHD are highly susceptible to "saccadic fatigue," the exhaustion caused by the eye constantly darting across a busy screen to parse disorganized information \cite{munoz2003saccadic}.
\textbf{ACESS Intervention:} The implementation of a strict "Focus Mode." This mode conditionally hides the primary navigation sidebar, the global footer, and all extraneous dashboard widgets when a learner enters a lesson. The content is restricted to a narrow, center-aligned column (maximum 65 characters per line), minimizing required eye movement and drastically reducing peripheral distraction. The technical implementation of Focus Mode, including the activity diagram and database schema, is detailed in Chapter 4 (see Figures~\ref{fig:act_educator} and Tables~\ref{tab:data_dict_22},~\ref{tab:data_dict_23}).

\subsubsection{Dyslexia}
Dyslexia involves difficulties with accurate or fluent word recognition and poor spelling and decoding abilities \cite{rello2013effect}. Research indicates that specific typographical adjustments can significantly improve reading velocity and comprehension \cite{franceschini_dyslexia}.
\textbf{ACESS Intervention:} The platform abandons reliance on browser-level zoom. Instead, it utilizes React Context to allow the user to select a "Dyslexia Profile." This injects CSS variables that alter the global font family to specialized typefaces designed to prevent letter mirroring (e.g., confusing 'b' and 'd'), increases the base line-height to 1.75, and manipulates letter-spacing (tracking) to prevent visual crowding \cite{batanero2014accessibility}. The Dyslexia Profile implementation, including dynamic CSS injection and preference storage, is detailed in Chapter 4 (see Figures~\ref{fig:act_student} and Tables~\ref{tab:data_dict_3},~\ref{tab:data_dict_2}).

\subsubsection{Autism Spectrum Disorder (ASD)}
Learners on the Autism Spectrum often thrive in highly structured, predictable environments and can experience severe distress when confronted with unexpected auditory or visual stimuli \cite{lawson2011autism}.
\textbf{ACESS Intervention:} The platform enforces strict structural consistency. Educators utilizing the Tiptap rich-text editor \cite{tiptap_docs} are prevented from creating chaotic, ad-hoc layouts. Furthermore, the system defaults to suppressing autoplaying media and allows the user to globally toggle a "Reduced Motion" state, ensuring the interface remains a calm, predictable schedule of events. The Reduced Motion and structural consistency features are implemented via the accessibility preference layer described in Chapter 4 (see Figures~\ref{fig:act_admin} and Tables~\ref{tab:data_dict_3}).

\subsection{Comparative Analysis of Existing LMS Platforms}

To justify the development of ACESS, an extensive evaluation of existing market-leading platforms was conducted. The platforms analyzed include Moodle \cite{moodle_software}, Canvas LMS \cite{canvas_lms}, Blackboard Learn \cite{blackboard_learn}, and Google Classroom \cite{google_classroom}. 

\subsubsection{Moodle}
Moodle is an open-source powerhouse, widely adopted globally \cite{moodle_architecture}. Its greatest strength is its modularity. However, this is also its critical failing regarding cognitive accessibility.
\textbf{Limitations:} Moodle's native interface is notoriously dense and text-heavy. While administrators can install accessibility plugins like the 'Accessibility block,' these are bolt-on solutions. They require the learner to actively engage with a separate menu to adjust settings, adding cognitive load. Furthermore, Moodle relies on educators to manually identify struggling students and assign remedial work. It lacks a native, real-time heuristic engine to automate this process.

\subsubsection{Canvas LMS}
Canvas by Instructure represents a more modern, streamlined UI compared to Moodle \cite{canvas_lms}. It utilizes a card-based dashboard that is visually appealing to neurotypical users.
\textbf{Limitations:} Canvas achieves basic WCAG compliance but fails to provide deep cognitive accommodations. It lacks native Text-to-Speech, forcing reliance on OS-level tools, and does not offer granular control over typography such as specific Dyslexia fonts or adjusted inter-letter spacing. The platform is highly linear. If a student fails a Canvas quiz, the system cannot automatically restructure the module path to enforce revision.

\subsubsection{Blackboard Learn}
Blackboard is an enterprise-grade solution focused heavily on synchronous learning integrations and comprehensive grading rubrics \cite{blackboard_learn}.
\textbf{Limitations:} Blackboard's interface is often criticized for complex, deeply nested navigation menus (often three or four clicks deep to access a specific assignment file). For learners with executive dysfunction (ADHD), this navigational friction is a massive barrier to task initiation. Like Moodle, Blackboard's accessibility features are largely reactive (e.g., Blackboard ALLY checks documents for screen-reader compatibility) rather than proactive (dynamically altering the interface for the user).

\subsubsection{Google Classroom}
Google Classroom is a stripped-down, lightweight system focused on integration with the Google Workspace ecosystem (Docs, Drive, Forms) \cite{google_classroom}.
\textbf{Limitations:} While its minimalist design reduces initial visual clutter, it is too rudimentary to function as a fully adaptive LMS. It completely lacks advanced features such as rule-based learning paths, granular progress telemetry (tracking time-spent-per-lesson), or integrated cognitive profiles. It functions more as an assignment drop-box than a pedagogical support engine.

\subsubsection{Alternative Approaches and Justification}

Several alternative technical approaches were evaluated during the design of the ACESS platform before arriving at the selected architecture:

\begin{itemize}
    \item \textbf{Monolithic Architecture vs. Decoupled Client-Server:} A monolithic Ruby on Rails or PHP Laravel application was considered for its simpler deployment model. However, this approach was rejected in favor of a decoupled Next.js frontend \cite{nextjs_ssr} with Supabase backend-as-a-service \cite{supabase_docs}, which allows the accessibility engine to dynamically inject CSS variables on the client without full page reloads, preserving the fluid user experience critical for ADHD learners.
    \item \textbf{CSS-in-JS Libraries vs. Tailwind CSS:} Alternatives such as styled-components and Emotion were evaluated. While they offer dynamic styling, they introduce runtime overhead that degrades performance on low-end devices. Tailwind CSS v4 \cite{tailwind_docs} with Just-in-Time compilation and CSS custom properties was selected because it generates minimal production CSS and enables instantaneous theme switching without JavaScript re-computation.
    \item \textbf{Document Database vs. Relational Database:} MongoDB was considered for its schema flexibility. However, the relational integrity enforced by PostgreSQL with foreign key constraints and Row-Level Security \cite{supabase_rls} was deemed essential for ensuring data consistency across enrollments, progress tracking, and quiz attempts, particularly given the complex relational model required by the adaptive recommendation engine.
    \item \textbf{Traditional State Management vs. React Context:} Redux was considered for global state management but was deemed overly verbose for the accessibility settings use case. React Context with the \texttt{AccessibilityProvider} pattern was chosen for its simplicity \cite{react_docs}, native React integration, and minimal boilerplate when propagating CSS theme attributes across the component tree.
\end{itemize}

\subsection{Project Methodology}

The ACESS platform was developed using the \textbf{Agile Scrum} methodology \cite{schwaber1995scrum}, an iterative framework that emphasizes continuous delivery, regular stakeholder feedback, and adaptive planning. Scrum was selected over traditional sequential models such as Waterfall because designing for cognitive accessibility is an inherently empirical process. Assumptions about the effectiveness of specific interface interventions, such as a Dyslexia font or Focus Mode layout, must be validated through iterative user testing rather than deferred to the final stages of development.

The development lifecycle was organized into six sprints, each spanning two weeks:

\begin{itemize}
    \item \textbf{Sprint 0: Requirements and Scaffolding.} Definition of the product backlog based on WCAG 2.2 guidelines and UDL principles. Establishment of the Next.js project structure, Supabase project initialization, and PostgreSQL schema design.
    \item \textbf{Sprints 1-2: Core Architecture.} Implementation of user authentication with Supabase Auth, database schema normalization with Row-Level Security policies, and basic role-based routing via middleware.
    \item \textbf{Sprints 3-4: Feature Development.} Development of the Educator authoring interface (Tiptap editor, course creation, quiz builder) and the Learner experience (course enrollment, lesson viewing, progress tracking).
    \item \textbf{Sprints 5-6: Advanced Features and Refinement.} Integration of the rule-based adaptive recommendation engine, accessibility personalization engine with CSS variable injection, analytics dashboards, certificate generation, and notification triggers.
\end{itemize}

Each sprint concluded with a review and retrospective, ensuring continuous alignment with project objectives. A detailed breakdown of the Agile implementation is provided in Chapter 3.

\subsection{Summary}

\subsection{Problem Analysis}

The ACESS platform is designed to address specific, systemic shortcomings in current educational technology. While the literature review in Chapter 2 established the general limitations of platforms such as Moodle, Canvas, Blackboard, and Google Classroom, this section analyzes the concrete operational problems that directly influenced the architectural decisions of ACESS.

\subsubsection{Current System Scenario}

Existing Learning Management Systems typically operate under a one-size-fits-all paradigm. Learners are presented with a fixed interface regardless of their cognitive profile, forcing neurodivergent users to seek third-party accommodations such as browser extensions for text-to-speech, external tools for font customization, or manual workarounds for managing visual clutter. This places the burden of accessibility on the learner rather than the system.

From a data flow perspective, current platforms process learner interactions in a linear fashion: enrollment, content delivery, assessment, and completion. They lack the feedback loops necessary to detect when a learner is struggling and respond adaptively. For example, if a student performs poorly on a quiz about database normalization, the system continues to present the next topic as though no intervention is needed.

\subsubsection{Detailed Problem Breakdown}

The problems identified in Chapter 1 are analyzed here in greater technical depth:

\begin{enumerate}
    \item \textbf{Rigid Typography and Layout:} Standard LMS platforms use fixed CSS stylesheets that do not respond to individual user needs. Learners with Dyslexia require specific typographic conditions, including increased letter spacing (tracking) of at least 0.12em, line height of 1.5 to 1.8, and sans-serif fonts such as Atkinson Hyperlegible \cite{atkinson2021atkinson} designed to prevent letter confusion, such as differentiating 'b' from 'd' \cite{batanero2014accessibility}. Without these adjustments, reading velocity decreases and cognitive fatigue accelerates, leading to early lesson abandonment.
    \item \textbf{Cognitive Overload from Navigation:} Platforms like Moodle and Blackboard present deeply nested navigation menus (often three to four levels deep) combined with dense dashboard widgets showing calendars, announcements, grades, and messages simultaneously. For learners with ADHD \cite{barkley1997adhd}, this multi-stimulus environment triggers rapid attention fragmentation, making it difficult to sustain focus on a single learning objective. The absence of a distraction-free mode means learners must manually close or minimize interface elements.
    \item \textbf{Lack of Adaptive Remediation:} Current assessment systems grade quizzes and record scores but do not analyze performance patterns to generate corrective action. When a learner fails a prerequisite concept, the system does not automatically unlock remedial content or recommend revision. This linear progression model assumes uniform learner readiness, which is particularly detrimental for neurodivergent students who may require repeated exposure to foundational topics before advancing.
\end{enumerate}

\subsection{Requirement Analysis}

\subsubsection{Data Requirement}

The ACESS platform requires a robust relational data model to manage the complex relationships between users, courses, lessons, assessments, and accessibility preferences. The system must store and process the following categories of data:

\begin{itemize}
    \item \textbf{User Data:} Authentication credentials (email, password hash), profile information (name, avatar, role), and notification preferences. User data must be isolated by role with strict access controls.
    \item \textbf{Accessibility Data:} Per-user cognitive profile settings including disability type, preferred font family and size, theme (light, dark, high-contrast, soft), line spacing, text-to-speech and caption toggles, reduced motion preference, simplified UI mode, and keyboard navigation flags. These settings must persist across sessions and apply globally to all interface components.
    \item \textbf{Curriculum Data:} Hierarchical course structures (courses, chapters, lessons) with rich-text HTML content, video assets, transcripts, and metadata such as difficulty level, estimated duration, and prerequisite relationships that drive the adaptive engine.
    \item \textbf{Assessment Data:} Quiz configurations (time limits, pass thresholds, max attempts), question banks with multiple-choice and scenario formats, answer options with correctness flags, and learner attempt records including scores and answer selections.
    \item \textbf{Telemetry Data:} Granular engagement metrics including lesson view status, time spent per lesson (in seconds), enrollment status (active, completed, dropped), and quiz attempt history with pass/fail outcomes.
    \item \textbf{Content and Achievement Data:} Interactive activity configurations (flashcards, drag-and-drop, memory games, timelines), certificate records with unique reference codes, and badge/achievement definitions and user progress toward them.
    \item \textbf{Notification and Communication Data:} System-generated notifications triggered by enrollment, quiz attempts, lesson additions, and course publications, along with lesson-level threaded comments and contact form submissions.
\end{itemize}

The detailed database schema and data dictionary for these entities are presented in Chapter 4, Section 4.3.

\subsubsection{Non-Functional Requirement}

Non-functional requirements (NFRs) specify the critical quality attributes and systemic constraints that govern the operation of the ACESS platform.

\paragraph{Performance}
\begin{itemize}
    \item \textbf{Page Response Time:} Next.js Client-side route transitions must execute within \textbf{300ms} to prevent cognitive disruption and maintain user focus.
    \item \textbf{Course Loading:} Server-side data fetching for initial course loads must resolve within \textbf{800ms}, leveraging edge caching where applicable.
    \item \textbf{Video Loading:} Embedded instructional videos must commence playback within \textbf{2 seconds} utilizing adaptive bitrate streaming via Supabase Storage CDNs.
    \item \textbf{Concurrent Users:} The PostgreSQL database, utilizing connection pooling (PgBouncer), must support a minimum of \textbf{500 concurrent} intensive operations (e.g., simultaneous quiz submissions) without degrading frontend TTFB (Time to First Byte).
\end{itemize}

\paragraph{Security}
\begin{itemize}
    \item \textbf{Authentication:} All user sessions must be cryptographically secured using JSON Web Tokens (JWT) managed by Supabase Auth, with a mandatory 15-minute inactivity timeout.
    \item \textbf{Authorization (RBAC):} The application middleware must definitively intercept and route requests based on the role claim within the JWT, achieving zero-trust client routing.
    \item \textbf{Database Protection:} Strict Row-Level Security (RLS) must be enabled on 100\% of tables containing Personally Identifiable Information (PII) or cognitive telemetry, ensuring isolation at the storage layer.
    \item \textbf{Input Validation:} All API endpoints must utilize Zod schema validation to sanitize incoming payloads, preventing SQL injection and Cross-Site Scripting (XSS).
\end{itemize}

\paragraph{Accessibility}
\begin{itemize}
    \item \textbf{WCAG Compliance:} The platform must strictly adhere to \textbf{WCAG 2.2 Level AA} standards \cite{wcag22}. Minimum contrast ratios must exceed 4.5:1 for standard text.
    \item \textbf{Dyslexia Support:} The system must natively provide the Atkinson Hyperlegible typeface \cite{atkinson2021atkinson} and allow users to dynamically increase line-spacing to 1.75em.
    \item \textbf{ADHD and Autism Support:} The implementation of a global reduced-motion attribute must programmatically disable all CSS and JavaScript animations \cite{framer_motion_docs}. Focus Mode must isolate content to a 65-character maximum width to reduce saccadic fatigue.
    \item \textbf{Screen Reader Compatibility:} All custom UI components must natively handle WAI-ARIA states such as aria-expanded and aria-hidden, and support complete keyboard navigation.
\end{itemize}

\paragraph{Usability}
\begin{itemize}
    \item \textbf{Consistent Navigation:} The primary navigational architecture must remain structurally identical across all modules to ensure predictability for Autistic learners.
    \item \textbf{Error Prevention:} Destructive actions (e.g., an Educator deleting a course) must require a confirmed multi-step verification modal to prevent accidental data loss.
\end{itemize}

\paragraph{Reliability}
\begin{itemize}
    \item \textbf{Data Integrity:} The database schema must strictly enforce foreign key constraints with cascade deletes where appropriate to prevent orphaned records, such as a lesson progress record existing without a valid enrollment.
    \item \textbf{Backup Strategy:} The Supabase infrastructure must be configured to execute automated, daily Point-in-Time Recovery (PITR) backups of the entire PostgreSQL volume.
\end{itemize}

\paragraph{Maintainability and Scalability}
\begin{itemize}
    \item \textbf{Modular Architecture:} The codebase must adhere to strict separation of concerns, decoupling the data-fetching logic (Server Components) from the interactive accessibility logic (Client Components).
    \item \textbf{Database Normalization:} The schema must adhere to the Third Normal Form (3NF) to eliminate data redundancy, ensuring the database scales efficiently as course and user volume increases.
\end{itemize}

\subsubsection{Others Requirement}

The software and hardware tools selected for the ACESS platform, as listed in Chapter 2, Section 2.4, are justified by the following architectural considerations:

\begin{itemize}
    \item \textbf{Next.js 16 and React \cite{nextjs_ssr}:} Server-side rendering and React Server Components reduce initial JavaScript payload, critical for rapid page loads that prevent cognitive drop-off for ADHD learners. The App Router provides middleware-based route protection that enforces role-based access at the edge.
    \item \textbf{Supabase with PostgreSQL \cite{supabase_docs}:} Row-Level Security provides data isolation \cite{supabase_rls} at the database level rather than relying solely on application-layer guards, ensuring that cognitive telemetry and assessment data remain cryptographically isolated per user.
    \item \textbf{Tailwind CSS v4 \cite{tailwind_docs}:} Just-in-Time compilation enables dynamic CSS variable injection, allowing instantaneous theme and typography switching without JavaScript re-evaluation, which is essential for real-time accessibility adjustments.
    \item \textbf{TypeScript \cite{typescript_docs}:} Static typing across the full stack reduces runtime errors in critical paths such as quiz scoring and accessibility profile application, where a type mismatch could result in incorrect interface rendering.
    \item \textbf{Vercel \cite{vercel_docs}:} Serverless edge deployment provides global low-latency access to the platform, which is particularly important for learners whose cognitive focus may be disrupted by slow page transitions.
\end{itemize}

    \item \textbf{Interactive Activity Module:} Governs the execution of varied assessment types, including standard multiple-choice quizzes and embedded video questions, immediately evaluating responses in the backend.
    \item \textbf{Analytics Module:} Aggregates data from lesson progress and quiz attempts to render visualizations, providing educators with actionable intelligence on cohort performance.
\end{itemize}

\subsubsection{User Interface Design}

The interface architecture of ACESS is organized into three role-specific shells, each with distinct navigation structures, input mechanisms, and output presentations.

\paragraph{Navigation Design}

The navigation architecture follows a role-based shell pattern enforced by Next.js middleware:

\begin{itemize}
    \item \textbf{Public Navigation:} Unauthenticated users access the landing page, login, signup, and password reset routes. The landing page features a hero section, feature highlights, system capabilities showcase, and a role-based demo modal.
    \item \textbf{Learner Navigation:} The learner shell (\texttt{LearnerShell}) contains a persistent collapsible sidebar with links to Dashboard, Courses, Progress, Certificates, Achievements, and Favorites. A top bar displays the accessibility settings toggle, notification bell with unread badge count, language switcher, and user profile menu. The sidebar auto-hides in Focus Mode when navigating to lesson routes.
    \item \textbf{Educator Navigation:} The educator shell (\texttt{EducatorShell}) provides a sidebar with Dashboard, My Courses, All Courses, Students, Analytics, and Certificates. A course workspace sub-navigation includes tabs for Builder, Lessons, Quiz Settings, Assets, and Preview.
    \item \textbf{Administrator Navigation:} The admin shell (\texttt{AdminShell}) includes links to Dashboard, Users, Courses, Certificates, Analytics, Reports, Instructor Applications, and Contact Messages, with CRUD sub-navigation for course and user management.
\end{itemize}

\paragraph{Input Design}

The system implements structured input forms with client-side and server-side validation. Key input interfaces include:

\begin{itemize}
    \item \textbf{User Registration and Profile:} Registration form collects email, password, and role selection (learner/educator) with Zod validation for email format and password strength (minimum 8 characters, uppercase, number). Profile editing allows avatar upload via Supabase Storage with MIME type validation.
    \item \textbf{Course Creation (Educator):} Course metadata form includes title (required, max 200 characters), slug (auto-generated with uniqueness validation), description, difficulty level (select: beginner/intermediate/advanced), category, and thumbnail upload. Validation prevents duplicate slugs at the database level.
    \item \textbf{Lesson Editor (Tiptap):} Rich-text editor with constrained toolbar (headings, bold, italic, lists, links, code blocks, tables). Accessibility validation enforces heading hierarchy (no skipped levels), alt-text requirements on images, and caption requirements on video embeds. Content is sanitized server-side before persisting to \texttt{lessons.content\_html}.
    \item \textbf{Quiz Builder (Educator):} Question creation interface supports multiple-choice and scenario types. Each question requires a minimum of two options with exactly one marked correct via radio button. Validation prevents publishing quizzes with unanswered questions or missing correct-option flags.
    \item \textbf{Accessibility Settings (Learner):} A modal interface with toggle switches, dropdown selects, and slider controls for font size, line spacing, theme (light/dark/soft/high\_contrast), background tint picker, TTS enablement, captions, reduced motion, simplified UI, and keyboard navigation. All changes apply instantly via React Context without page reload.
    \item \textbf{Interactive Activity Builder (Educator):} Form-based interfaces for creating flashcards (front/back text or image), drag-and-drop activities (categories and draggable items), fill-in-the-blanks (sentence with missing word), memory games (card pairs), and timelines (ordered events). Each activity type has dedicated validation rules.
\end{itemize}

\paragraph{Output Design}

The system produces the following outputs:

\begin{itemize}
    \item \textbf{Progress Analytics (Recharts):} Educator dashboards render bar charts, line charts, and pie charts using Recharts components, visualizing enrollment rates, lesson completion percentages, average time spent per lesson, and quiz score distributions. Data is aggregated server-side via PostgreSQL queries and delivered as typed arrays to chart components.
    \item \textbf{PDF Certificates (jsPDF):} Upon course completion, the system generates a landscape PDF certificate containing the learner's full name, course title, completion date, a unique alphanumeric reference code, and institutional branding. The PDF is generated server-side via \texttt{generatePDFCertificate()} in \texttt{certificate-utils.ts} and uploaded to Supabase Storage with the URL stored in the \texttt{certificates} table.
    \item \textbf{Administrator Reports:} Custom report generation with configurable date ranges, user filters, and metric selection. Output is available as PDF or CSV download, generated via \texttt{/api/admin/reports}.
    \item \textbf{Badge and Achievement Notifications:} Visual milestone badges are awarded and displayed on the learner dashboard and certificates page when predefined criteria (e.g., 100\% module completion, perfect quiz score) are met, evaluated by the \texttt{achievement-engine.ts}.
    \item \textbf{Progress Summary:} The learner dashboard displays a progress overview card showing enrolled course count, completed lessons, certificates earned, and achievements unlocked, with a call-to-action for the next recommended lesson.
\end{itemize}

This domain manages authentication, role assignment, and per-user accessibility state.

This domain tracks all learning content structure and learner engagement telemetry.
    
    \item \textbf{Content \& Media Domain:} Encompassing \texttt{lesson\_comments}, \texttt{lesson\_summaries}, \texttt{lesson\_versions}, \texttt{lesson\_interactive\_content}, \texttt{lesson\_templates}, \texttt{media\_assets}, \texttt{notifications}, \texttt{contact\_messages}, \texttt{certificates}, \texttt{certificate\_templates}, and \texttt{certificate\_verifications}. This domain manages content artefacts, communication, and certification.
\end{enumerate}

\subsection{Detailed Design}

\subsubsection{Software Design}

The ACESS platform follows a component-based architecture using React Server Components (RSC) and Client Components \cite{react_docs}, organized by feature domain. The key software components are described below.

\paragraph{Provider Components (Global State)}

\begin{itemize}
    \item \textbf{AuthProvider:} Wraps the application root. Initializes the Supabase client session via \texttt{@supabase/ssr}, exposes \texttt{user} object, \texttt{session}, \texttt{signIn()}, \texttt{signUp()}, \texttt{signOut()} through React Context. On mount, fetches the user's role and profile metadata. Responsible for session refresh and 15-minute inactivity timeout detection via \texttt{SessionTimeout} component.
    \item \textbf{AccessibilityProvider:} Loads \texttt{user\_accessibility\_settings} from Supabase on authentication. Exposes setting values and a \texttt{updateSetting()} method through context. On every setting change, writes to the database and applies CSS \texttt{data-*} attributes to the \texttt{<html>} element (e.g., \texttt{data-theme="dyslexia"}, \texttt{data-reduced-motion="true"}) which Tailwind CSS v4 \cite{tailwind_docs} intercepts to re-theme the entire component tree.
    \item \textbf{LanguageProvider:} Loads the user's \texttt{preferred\_language} from profile settings, provides a \texttt{t()} function that maps keys to locale strings in \texttt{src/locales/en.ts} and \texttt{ms.ts}. Supports runtime language switching without page reload.
\end{itemize}

\paragraph{Learner Components}

\begin{itemize}
    \item \textbf{LearnerDashboard:} Server component that fetches enrolled courses, recommendations, recent progress, and achievement summary. Renders \texttt{WelcomeSection}, \texttt{MyCoursesSection}, \texttt{ProgressOverview}, and \texttt{QuickActions} client sub-components for interactive elements.
    \item \textbf{LessonPage:} Client component receiving \texttt{lessonId} as prop. Fetches lesson content (\texttt{content\_html}), video URL, and quiz configuration. Renders sanitized HTML via \texttt{LessonRenderer}. In Focus Mode, conditionally omits sidebar and header. Tracks time spent via \texttt{setInterval} dispatching \texttt{PATCH /api/lesson/[id]/progress} every 30 seconds.
    \item \textbf{QuizPage:} Client component managing question navigation state, timer (derived from \texttt{quizzes.time\_limit\_seconds}), selected answers map, and auto-submit logic on timeout. On submission, sends \texttt{selected\_option\_id[]} to \texttt{POST /api/quiz/[id]/attempt} and displays \texttt{QuizResultModal} with score, pass/fail status, and answer review.
    \item \textbf{AccessibilitySettingsModal:} Client component with toggle switches and dropdowns for all a11y settings. Each input calls \texttt{updateSetting()} on the \texttt{AccessibilityProvider} context, providing real-time preview of changes.
\end{itemize}

\paragraph{Educator Components}

\begin{itemize}
    \item \textbf{CourseWorkspace:} Client component acting as the educator's main working area. Manages tab state (Builder, Lessons, Quiz Settings, Assets, Preview). Contains nested routing for lesson editing within the course context.
    \item \textbf{LessonEditor:} Client component wrapping the Tiptap editor instance \cite{tiptap_docs}. Configures the editor with extensions: StarterKit, Underline, Link, Image, Table, CodeBlockLowlight, TextAlign, Highlight, TextStyle, Color, and Placeholder. Strips disallowed tags on save. Calls \texttt{PUT /api/educator/lessons/[id]} to persist the sanitized HTML.
    \item \textbf{QuizBuilderModal:} Client component for creating and editing quiz questions. Manages a dynamic list of questions each with a type selector (multiple-choice or scenario), question text input, and a dynamic list of option rows with radio-button correct-flag selection. Validates that each question has at least one correct option before allowing save.
    \item \textbf{InteractiveActivityBuilder:} Client component with sub-builders for each activity type: \texttt{FlashcardBuilder}, \texttt{DragDropBuilder}, \texttt{FillBlanksBuilder}, \texttt{MemoryGameBuilder}, \texttt{TimelineBuilder}. Each sub-builder manages its specific data structure and emits a typed JSON object on save.
\end{itemize}

\paragraph{Administrator Components}

\begin{itemize}
    \item \textbf{UserManagement:} Client component with a plain HTML table for sorting, filtering, and paginating the users list. Supports role reassignment via \texttt{RoleEditModal} and soft-delete toggling.
    \item \textbf{CourseManagement:} Client component listing all courses with status filters (draft, pending\_review, published). Includes \texttt{ThumbnailPicker} for assigning stock images and \texttt{AdminCourseSettingsTab} for system-wide course configuration.
    \item \textbf{AnalyticsDashboard:} Client/server hybrid component. Server-fetches aggregated platform metrics (total users, enrollments, active courses, completion rates). Renders custom CSS-based bar charts with interactive tooltips and date range filtering.
\end{itemize}

\paragraph{Core API Routes and Server Logic}

\begin{itemize}
    \item \textbf{POST /api/recommendations/generate:} Invoked server-side after quiz attempt evaluation. Executes the \texttt{recommendation-engine.ts} which analyzes the learner's enrollment context, quiz score, and prerequisite lesson chain to generate up to three recommendations (revision, standard, advanced, or cross-course). Uses Supabase service role to bypass RLS for cross-user queries.
    \item \textbf{PUT /api/educator/lessons/[id]:} Receives sanitized HTML from Tiptap, validates with Zod, updates the \texttt{lessons} table, and creates a version snapshot in \texttt{lesson\_versions} for rollback support.
    \item \textbf{POST /api/certificates/custom:} Receives a custom PDF certificate file from the educator, validates file type (PDF only) and size (max 5MB), uploads to Supabase Storage \texttt{course-assets} bucket, and stores the URL reference.
    \item \textbf{Recommendation Engine (\texttt{recommendation-engine.ts}):} Contains \texttt{generateRecommendations()} function. Logic branches: revision (if recent quiz failed), standard (if unviewed lessons remain), advanced (if near course completion), cross-course (if tag-matched courses exist). Returns an ordered array of recommended lesson IDs with trigger reasons.
    \item \textbf{Achievement Engine (\texttt{achievement-engine.ts}):} Contains \texttt{evaluateAchievements()} function called after quiz attempts and lesson completions. Checks learner's current progress against \texttt{course\_achievements} criteria and grants new badges via \texttt{user\_achievements} insertion.
    \item \textbf{Adaptive Engine (\texttt{adaptive-engine.ts}):} Applies preset-based accessibility configuration (Dyslexia, ADHD, Autism, Dyscalculia) with age-adapted sub-variants. Each preset defines a combination of settings for font family, spacing, theme, tint, reading spotlight, focus mode, chunked content, TTS, and simplified UI, applied via CSS data attributes.
\end{itemize}

\paragraph{Recommendation Engine Pseudocode}

Algorithm~\ref{alg:recommendation} presents the pseudocode for the \texttt{generateRecommendations()} function, illustrating the rule-based branching logic used to generate personalized lesson recommendations.

\begin{algorithm}[H]
\caption{generateRecommendations()}
\label{alg:recommendation}
\begin{algorithmic}[1]
\Require $learner\_id$, $course\_id$, $quiz\_score$
\Ensure Ordered list of recommended lesson IDs with reasons
\State $enrolled \gets$ \textsc{checkEnrollment}($learner\_id$, $course\_id$)
\State $recent\_quiz \gets$ \textsc{getRecentQuizAttempt}($learner\_id$, $course\_id$)
\State $viewed \gets$ \textsc{getViewedLessons}($learner\_id$, $course\_id$)
\State $all\_lessons \gets$ \textsc{getCourseLessons}($course\_id$)
\State $recommendations \gets \emptyset$

\If{$recent\_quiz.score < passing\_threshold$}
    \State $failing\_lesson \gets recent\_quiz.lesson\_id$
    \State \textsc{append}($recommendations$, \{revision, $failing\_lesson$\})
\EndIf

\ForAll{$lesson$ in $all\_lessons \setminus viewed$}
    \State \textsc{append}($recommendations$, \{standard, $lesson$\})
    \If{$|recommendations| \geq 3$}
        \State \textbf{break}
    \EndIf
\EndFor

\If{course\_completion\_pct $\geq 80\%$}
    \State $next\_course \gets$ \textsc{getTagMatchedCourse}($course\_id$)
    \State \textsc{append}($recommendations$, \{cross-course, $next\_course$\})
\EndIf

\State \Return $recommendations$
\end{algorithmic}
\end{algorithm}

The ACESS database is implemented on Supabase PostgreSQL 15 \cite{supabase_docs}. The physical schema enforces relational integrity through foreign keys, unique constraints, and Row-Level Security policies.

\paragraph{Row-Level Security Policies}

RLS is enabled on all tables containing user data. Representative policies include:

\begin{tcolorbox}[listing only, breakable, boxsep=2pt, left=2pt, right=2pt]
\footnotesize\ttfamily
-- Learners can only read their own lesson progress
CREATE POLICY lesson\_progress\_learner\_select
ON lesson\_progress FOR SELECT
USING (auth.uid() IN (
    SELECT user\_id FROM enrollments WHERE id = enrollment\_id
));

-- Educators can CRUD their own courses
CREATE POLICY courses\_educator\_insert
ON courses FOR INSERT
WITH CHECK (auth.uid() = created\_by);

-- Admins can read all rows
CREATE POLICY users\_admin\_select
ON users FOR SELECT
USING (auth.role() = 'admin');
\end{tcolorbox}

\paragraph{Database Triggers}

Five notification triggers are implemented as PL/pgSQL functions attached to INSERT and UPDATE events:

\begin{tcolorbox}[listing only, breakable, boxsep=2pt, left=2pt, right=2pt]
\footnotesize\ttfamily
CREATE FUNCTION notify\_on\_quiz\_attempt()
RETURNS TRIGGER AS \$\$
BEGIN
    IF NEW.score\_pct < (SELECT pass\_threshold\_pct
                         FROM quizzes WHERE id = NEW.quiz\_id) THEN
        INSERT INTO notifications (user\_id, type, title, body, metadata)
        VALUES (
            (SELECT created\_by FROM courses WHERE id = (
                SELECT course\_id FROM enrollments WHERE id = NEW.enrollment\_id
            )),
            'quiz\_failed',
            'Student Failed Quiz',
            'A learner scored below the passing threshold.',
            jsonb\_build\_object('attempt\_id', NEW.id, 'score', NEW.score\_pct)
        );
    END IF;
    RETURN NEW;
END;
\$\$ LANGUAGE plpgsql;

CREATE TRIGGER notify\_on\_quiz\_attempt\_trigger
AFTER INSERT ON quiz\_attempts
FOR EACH ROW EXECUTE FUNCTION notify\_on\_quiz\_attempt();
\end{tcolorbox}
\subsubsection{Systemic Implementation Details}

\paragraph{Interactive Video Questions}
To prevent passive consumption, ACESS implements embedded video questions. This is achieved by creating a custom React hook \texttt{useVideoTime()} connected to an HTML5 video player reference. An array of timestamp objects is loaded from the database. When the video's current time intersects a target timestamp, the state \texttt{isVideoPaused} is forced to true, and a z-index overlay modal is mounted containing a specific \texttt{quiz\_question}. The video cannot be resumed until the user interacts with the prompt.

\paragraph{Database Trigger Architecture (Notifications)}
To maintain a decoupled architecture and prevent the Next.js API routes from becoming bloated with notification logic, ACESS utilizes native PostgreSQL Triggers. For example, the \texttt{notify\_on\_quiz\_attempt} trigger is attached to the \texttt{quiz\_attempts} table. Upon an \texttt{INSERT} where \texttt{score\_pct < pass\_threshold}, the trigger executes a PL/pgSQL function that automatically formats a payload and executes an \texttt{INSERT} into the \texttt{notifications} table, targeting the \texttt{created\_by} ID of the parent course, instantly alerting the educator of the failure via real-time database subscription.

\paragraph{Session Timeout and Security}
A \texttt{SessionTimeout} component wraps authenticated routes. It tracks user activity via \texttt{mousemove} and \texttt{keydown} event listeners. After 15 minutes of inactivity, a warning modal appears with a 30-second countdown. If no response is received, the user is forcibly signed out via \texttt{supabase.auth.signOut()} and redirected to the login page.

\subsection{Limitations of Design}
\label{sec:design-limitations}

The current ACESS design has several acknowledged limitations that define the boundary of this implementation:

\begin{itemize}
    \item \textbf{Content scope:} The system currently supports only text-based lessons with embedded video. Audio-only content, interactive simulations, and downloadable PDF resources are not yet supported.
    
    \item \textbf{Adaptive engine:} The rule-based recommendation engine operates on quiz performance only. It does not incorporate learner behaviour analytics such as time-on-task or scroll patterns, nor does it utilize real-time physiological signals.
    
    \item \textbf{Expert review only:} Accessibility evaluation was conducted through structured walkthroughs and expert review against WCAG 2.1 Level AA criteria. Full user testing with neurodivergent participants was beyond the scope of this project phase and is recommended for future work.
    
    \item \textbf{Single-language support:} The platform interface and content are English-only. Multilingual and right-to-left language support has not been implemented.
    
    \item \textbf{Offline capability:} All features require an active internet connection. No offline content caching or progressive web app (PWA) functionality has been implemented.
\end{itemize}
